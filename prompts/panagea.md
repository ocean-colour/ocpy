#### Column conventions

- **Common metadata columns** (first four of every file): `ID (idx)`,
  `Date/Time` (ISO 8601, e.g. `1997-01-02T12:48`), `Latitude`, `Longitude`.
  Most files also carry `Depth water [m]` (from ETOPO1, clamped to â‰¤0) and a
  set of provenance `Comment` columns (dataset / subdataset / contributor)
  plus quality flags (`QF time`, `QF Chl`).
- **Native-wavelength files** name spectral columns like
  `RRS [1/sr] (at 313nm)` or `APH [1/m] (at 290.2nm)` â€” hundreds of
  closely-spaced wavelengths (hence the very wide IOP/Rrs files).
- **Sat-band files** name columns in pairs: the value at a sensor band, e.g.
  `RRS [1/sr] (at +-2nm of OLCI-S3A band 11)`, followed by the actual
  `Lambda [nm] (assigned to ... band 11)`. Sensors covered: SeaWiFS,
  MODIS-AQUA, VIIRS, OLCI (S3A/S3B) and MERIS.
- **IOP variables** in `iopskdtsm`: `APH` (algal pigment absorption),
  `acCDOM` (CDOM + detrital absorption), `bbp` (particulate backscatter),
  `KD` (diffuse attenuation), `TSS` (total suspended matter / TSM).

#### Joining across files

`ID` is a global observation index shared across all 7 files (it is not
contiguous within a file â€” it ranges 1..151,673 with gaps). The same `ID`
in two files refers to the same in-situ observation, so files can be merged
on `ID`. Overlap is partial (e.g. chla âˆ© rrs_satbands2 = 3,840 IDs), since
not every observation has every measurement type.

#### Implications for the API

- A single loader can serve all 7 files: skip the right number of header
  lines (or detect the `*/` line automatically), then parse the column line.
- The header `Parameter(s):` block is a machine-readable column dictionary
  (long name, unit, PI, method, comment) that could be parsed to build
  tidy/renamed columns and per-column metadata.
- For the wide spectral files, a "wide â†’ long" reshape (ID, wavelength,
  value) is likely the most useful in-memory form for downstream analysis.

#### Resolved design decisions (from Q&A, 2026-06-15)

These reflect the user's answers and now drive the plan:

- **All 7 files.** One generic loader handles every `.tab` file.
- **Auto-detect the header.** The loader scans for the line ending the
  `/* ... */` block (the `*/` line) and treats the following line as the
  column header â€” no hard-coded `skiprows` per file. This survives any
  re-download.
- **Primary use case = generating individual Rrs (and IOP) spectra.** The
  driving activity is: given an observation, produce its spectrum â€” a set of
  (wavelength, value) pairs. So the design is:
  1. Load each file faithfully into a wide DataFrame indexed by `ID`.
  2. Parse the wavelength out of each spectral column name (e.g.
     `RRS [1/sr] (at 349.01nm)` â†’ `349.01` as a float; sat-band columns
     labelled by their paired `Lambda [nm]` value).
  3. Provide a helper that, for a given `ID`, returns that observation's
     spectrum as a wavelength-indexed `pandas.Series` with missing
     wavelengths dropped â€” i.e. the natural input for plotting/fitting a
     single Rrs spectrum. A bulk long-format reshape (ID Ã— wavelength Ã—
     value) is offered as a secondary convenience for cross-observation
     analysis.
  This keeps the wide table available (faithful, fast `ID` lookup) while
  making single-spectrum extraction the easy path.
