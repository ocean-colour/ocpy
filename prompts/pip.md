## Q&A

**Q1 (RESOLVED): PyPI distribution name.** The bare name `ocpy` is already
taken on PyPI by an unrelated project (documented in the sibling bing
repo's pyproject.toml). Per your answer, the distribution is named
**`ocpy-ocean`**; the import package stays `ocpy`. Verify `ocpy-ocean` is
free on PyPI before the first upload.

**Q2 (RELEASE BLOCKER -- needs decision): stale `oceancolor` imports.**
The package was renamed from `oceancolor` to `ocpy`, but several modules
and tests still import the old name and will break on a fresh
`pip install` (the old package won't exist). The CI workflow already
limits its test scope for exactly this reason. Affected:
- `ocpy/tara/ingest.py`  -- `from oceancolor.tara import io`,
  `resource_filename('oceancolor', 'data')`
- `ocpy/ls2/kd_nn.py`    -- `from oceancolor.ls2 import io`
- `ocpy/ls2/io.py`       -- 4x `resource_filename('oceancolor', ...)`
- `ocpy/polarize/load_data.py` -- 2x `resource_filename('oceancolor', ...)`
- tests: `test_ls2.py`, `test_water.py`, `test_ph.py`,
  `test_polarize.py`, `test_ls2_kd.py`

I left these untouched -- the migration is a separate, broader effort
already tracked in CI, not a packaging change. Recommend completing it
(or shipping a temporary `oceancolor` -> `ocpy` alias) before release.

A. Replace all oceancolor imports with ocpy.
