  - The wavelengths

  - The errors
  - The units
  - The metadata
  - The source of the data
  - The date of the data
  - The location of the data
  - The depth of the data
  - The quality of the data

These items should be basic Python objects (numpy arrays, floats, etc.).

We then want methods that act on them.  Whereever possible, these methods will operate on basic Python objects (numpy arrays, floats, etc.) rather than classes.

### Findings

#### How spectra are handled today

There is no single, general spectrum abstraction in `ocpy`. Spectra are
handled ad hoc, per data source, almost always as bare numpy arrays plus
loosely-associated wavelength arrays:

- **`ocpy/utils/spectra.py`** â€” two free functions only, no class:
  - `rebin(wv_nm, values, wv_grid, err_vals=None)` â€” interpolates a single
    spectrum (and optional errors) onto a new grid via `scipy.interp1d`,
    `fill_value=np.nan` outside bounds. Returns `(new_values, new_err)`.
  - `rebin_to_grid(wv_nm, values, err_vals, wv_grid)` â€” nearest-neighbour
    bin averaging (no interpolation) for a *stack* of spectra. Note the
    shape convention is awkward/inconsistent: the docstring says values are
    `(nwave, nspec)` but the loop indexes `values.shape[1]` as the spectrum
    count and returns `(nspec, nwv)`. Worth fixing in the refactor.
  - Both operate purely on numpy arrays â€” already aligned with the
    "methods over classes, basic objects" requirement.

- **`ocpy/tara/spectra.py`** â€” Tara-specific extraction: `parse_wavelengths`
  pulls wavelengths out of column-name strings (e.g. `ap440`), and
  `spectbl_from_keys` returns `(values, err)` arrays of shape
  `(len(keys), len(tbl))`. Wavelength/value/error are kept as separate,
  caller-managed arrays. `tara/explore.py` then calls
  `utils.spectra.rebin_to_grid` on those arrays.

- **`ocpy/insitu/panagea.py`** â€” the most mature representation. A loaded
  dataset is a `pandas.DataFrame` indexed by observation `ID`, with rich
  per-column metadata stashed in `df.attrs['columns']` (orig name, unit,
  parsed wavelength, sensor/band, role, long_name). `spectrum(df, obs_id,
  kind='rrs')` returns a `pandas.Series` of values indexed by wavelength
  (sorted, NaNs dropped). Handles both native-wavelength and satellite-band
  layouts. Carries unit and metadata, but errors are *not* paired with
  values, and per-observation provenance (date, lat/lon, depth, QF) lives in
  separate DataFrame columns rather than travelling with an extracted
  spectrum.

- **`ocpy/hydrolight/loisel23.py`** â€” Loisel+2023 Hydrolight outputs are
  `xarray.Dataset`s with a `Lambda` coordinate and variables `aph`, `bb`,
  etc., shape `(nspec, nwave)`. `calc_Chl` indexes `aph` at 440 nm. No unit
  metadata exposed; wavelength is shared across all spectra in a file.

#### Open discussion A â€” representation: xarray vs. lightweight class (Q3)

The user is weighing using an `xarray.Dataset` as *the* spectrum
representation (as Loisel23 already does). Pros and cons:

**Pros of an xarray Dataset**
- Native to Loisel23 â†’ zero-friction adaptation for that source.
- Self-describing: a named `wavelength` coordinate indexes the values, much
  like PANAGEA's wavelength-indexed Series.
- `.attrs` (dataset- and variable-level) maps cleanly onto the requested
  provenance fields (units, source, date, lat, lon, depth, quality).
- Values + errors sit as two data variables sharing the wavelength coord.
- Free, well-tested machinery: `.interp`, `.sel`, automatic coordinate
  alignment/broadcasting, NaN handling, and netCDF serialization.
- The gridded-stack case (Q4) is natural: a 2D `(spectrum, wavelength)` data
  variable over a shared wavelength coord.

**Cons of an xarray Dataset**
- It is **not** a "basic Python object" â€” this is in direct tension with the
  Requirements ("very lightweight class... numpy arrays, floats, etc." and
  "methods... operate on basic Python objects... rather than classes").
- The **ragged-stack case** (different/different-length wavelength axes, the
  hard half of Q4) does not fit one Dataset cleanly: xarray aligns on the
  coordinate by NaN-filling the union grid, which wastes memory and conflates
  "not measured here" with "missing".
- `.attrs` must be netCDF-serializable to keep the I/O benefit, restricting
  what can be stashed (no arbitrary Python objects); attrs are untyped.
- Heavyweight and verbose for what is, for a single spectrum, a few arrays
  plus a handful of scalars.

**Recommendation (middle path):** keep the lightweight class (numpy arrays +
scalars) as the canonical in-memory object â€” honoring the "basic objects"
requirement and keeping the array-level methods in `ocpy/spectra/utils.py`
operating on plain numpy â€” and add thin `to_xarray()` / `from_xarray()`
adapters. That gives Loisel23 interop and netCDF I/O for free without paying
the xarray tax everywhere, and the ragged case stays expressible. See Q&A
round 2, Q9.

#### Open discussion B â€” single vs. stacked spectra (Q4)

The user notes: same wavelengths â†’ can stack; different (and possibly
different-length) wavelengths â†’ must hold separately. Likely two objects.

Proposed shape:
- **`Spectrum`** â€” exactly one spectrum: 1D `wavelength`, 1D `values`, 1D
  `errors`, plus the scalar provenance fields.
- **`SpectrumStack`** â€” a collection, backed by a **list of `Spectrum`** so
  ragged members (different grids/sizes) are always representable. It offers
  a fast-path **gridded** view â€” a 2D `(nspec, nwave)` array over one shared
  wavelength axis with per-spectrum provenance arrays â€” available when all
  members already share a grid, or after `rebin`/`align_to_grid` resamples
  them onto a common grid. So: ragged list is the general store; the 2D
  array is an optimization unlocked by a shared grid.

This cleanly separates the easy gridded case (Loisel23, rebinned Tara) from
the ragged case (raw PANAGEA/Tara with heterogeneous bands). See Q&A round 2,
Q10â€“Q12.

#### Other implications (unchanged)

- Operations (rebin, interpolate, slice-by-wavelength, value-at-wavelength,
  arithmetic) stay as free functions on numpy arrays in
  `ocpy/spectra/utils.py`, with thin class methods that delegate â€” extending
  the existing style.
- The class should expose adapters from a PANAGEA row, a Loisel23 Dataset
  row, and a Tara table row so the three sources converge.
